# Amplifier Live Control Center — Design Spec

**Date:** 2026-03-20
**Status:** Ready for /create-plan (next session)

## Problem

The Amplifier docs site is 26 static HTML pages with baked-in sidebar menus and sample data. Adding a page requires regenerating all pages to update the sidebar. The dashboard has no live data. Docs and metrics are disconnected. No API for LLMs or external tools.

## Goal

Build a hybrid system: static docs (kept, they work) + FastAPI sidecar serving live dashboard data, shared sidebar, and LLM-ready API endpoints.

## Architecture

### How it works

1. **Static docs pages** stay as-is (generated by `generate.py`), but sidebar `<nav>` replaced with:
   ```html
   <nav id="sidebar"></nav>
   <script>
   fetch('/api/sidebar').then(r => r.text()).then(h => {
     document.getElementById('sidebar').innerHTML = h;
     // Highlight active page
     const active = document.querySelector(`a[href="${location.pathname}"]`);
     if (active) active.classList.add('active');
   });
   </script>
   ```

2. **FastAPI sidecar** (`docs-server.py`) runs alongside IIS:
   - `/api/sidebar` — renders sidebar HTML from `manifest.json` page list (single source of truth)
   - `/api/metrics` — reads JSONL files, returns aggregated JSON (reviews, failures, velocity)
   - `/api/status` — current system status (git branch, recent commits, rate limit state)
   - `/api/docs-index` — page list with titles, last-updated, source files
   - `/ws/events` — WebSocket for live dashboard updates (push when JSONL files change)
   - `/llms.txt`, `/llms-full.txt`, `/llms-structured.json` — dynamic, always current

3. **Dashboard page** (`/en/dashboard.html`) uses `fetch('/api/metrics')` for live data instead of baked-in samples. Auto-refreshes every 60s or via WebSocket.

4. **IIS routing:**
   - `/en/*`, `/pl/*` → static files (existing)
   - `/api/*`, `/ws/*`, `/llms*` → reverse proxy to FastAPI on localhost:8099
   - `/` → redirect to `/en/dashboard.html`

### Sidebar: Single Source of Truth

The sidebar is generated from `manifest.json` by FastAPI. When a page is added to manifest, the sidebar updates everywhere instantly — no regeneration of all pages needed.

The sidebar endpoint returns HTML matching the existing design system CSS classes (`site-sidebar`, `nav-link`, `nav-section`, etc.).

### Data Sources for /api/metrics

| Metric | Source | Update frequency |
|--------|--------|-----------------|
| Review stats | `${CLAUDE_PLUGIN_DATA}/reviews/history.jsonl` | After each review |
| Failures | `${CLAUDE_PLUGIN_DATA}/failures.jsonl` | On API errors |
| Rate limits | `${CLAUDE_PLUGIN_DATA}/rate-limit-flag.json` | On rate limit events |
| Eval scores | `${CLAUDE_PLUGIN_DATA}/eval-history.jsonl` | After /evaluate |
| Git velocity | `git log` (live) | Real-time |
| Session count | Recall index SQLite | Periodic |

### Tech Stack

- **FastAPI** + **uvicorn** (Python, UV-managed)
- **watchfiles** for JSONL file watching → WebSocket push
- **IIS** reverse proxy (ARR module, already installed on DEV)
- Existing design system CSS (no new frontend framework)

## Rollout Plan

1. Build FastAPI sidecar with `/api/sidebar` and `/api/metrics`
2. Create `scripts/update-sidebar.py` — regex-replace inline sidebars in all 26 pages with fetch-based loader
3. Run migration on existing pages
4. Configure IIS reverse proxy for `/api/*` → localhost:8099
5. Update `generate.py` to output fetch-based sidebar in new pages (not inline)
6. Add WebSocket live updates to dashboard
7. Deploy as Windows service (or scheduled task)
