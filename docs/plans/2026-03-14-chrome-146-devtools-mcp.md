# Chrome 146 DevTools MCP Integration Plan

> **For Claude:** REQUIRED: Use /subagent-dev to implement this plan. Each task specifies its Agent — dispatch that Amplifier agent as the subagent for implementation. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Integrate Chrome DevTools MCP Server (v0.20.0) as a new MCP server for Lighthouse audits, accessibility checks, and performance profiling. Update /browser-audit and /visual to leverage Chrome 146 web platform features.

**Architecture:** Add `chrome-devtools-mcp` alongside existing Playwright MCP (not replacing it). Playwright handles interactive automation (click, type, forms). DevTools MCP handles auditing (Lighthouse, a11y, performance). Both coexist — different tools for different jobs.

**Tech Stack:** chrome-devtools-mcp v0.20.0, Chrome 146, CSS scroll-driven animations

### Task 1: Register chrome-devtools-mcp in platform configs

**Agent:** modular-builder

**Files:**

- Modify: `config/platform/mcp.linux.json`
- Modify: `scripts/setup-platform-config.sh` — add `chrome-devtools` to enabled list
- Modify: `.claude/settings.json` — regenerated by setup script

- [ ] **Step 1: Add chrome-devtools-mcp to Windows MCP config**

In `config/platform/mcp.windows.json`, add after the `autocontext` entry:

```json
    "chrome-devtools": {
      "command": "cmd",
      "args": ["/c", "npx", "chrome-devtools-mcp@latest", "--acceptInsecureCerts", "--headless"],
      "disabled": true
    }
```

Note: `--acceptInsecureCerts` for our self-signed `*.ergonet.pl` cert. `--headless` for server environment. `disabled: true` — opt-in per session, not always-on (it launches a Chrome instance).

- [ ] **Step 2: Add chrome-devtools-mcp to Linux MCP config**

### Task 3: Add Chrome 146 web platform features to /visual generated pages

**Agent:** modular-builder

**Files:**
- Modify: `.claude/commands/visual.md` — or the visual-explainer skill that generates HTML pages

The visual-explainer skill generates self-contained HTML pages. We want to add Chrome 146 CSS features that enhance the generated output.

- [ ] **Step 1: Identify the CSS template location**

The visual-explainer skill generates inline CSS in HTML pages. Find where the CSS patterns are defined:

```bash
grep -rn "scroll-animation\|@keyframes\|animation:" .claude/commands/visual.md | head -5
```

Also check if there's a shared CSS patterns file:
```bash
find . -name "css-patterns*" -o -name "visual-template*" 2>/dev/null
```

- [ ] **Step 2: Add scroll-triggered animation CSS to the visual template guidance**

In the visual-explainer or project-recap skill template, add this CSS pattern to the recommended styles:

```css
/* Chrome 146: Scroll-triggered animations — sections fade in on scroll */
@supports (animation-timeline: scroll()) {
  .card, section, .timeline-item {
    opacity: 0;
    transform: translateY(20px);
    animation: fadeInUp 0.6s ease forwards;
    animation-timeline: view();
    animation-range: entry 0% entry 100%;
  }
  @keyframes fadeInUp {
    to { opacity: 1; transform: translateY(0); }
  }
}
```

The `@supports` block ensures graceful degradation on older browsers.

- [ ] **Step 3: Add text-scale meta tag to generated HTML pages**

Add to the template's `<head>` section:

```html
<meta name="text-scale" content="auto">
```

This tells Chrome 146 to honor OS-level text scaling preferences instead of applying browser autosizing heuristics.
